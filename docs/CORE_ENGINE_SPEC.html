Core Engine Specification

This document outlines the high-level architecture, module interactions, and the primary User Experience event loop for Does It Glider.

1. The User Experience Event Loop

The core loop is driven by the browser's native requestAnimationFrame. It ensures smooth 60FPS rendering (when possible) by calculating the Life rules and syncing them to the DOM exactly when the browser is ready to paint.

Loops indefinitely until pattern stabilizes or user provides new seed.

2. Full Architecture & Data Flow

The application strictly separates logic from presentation. The DOM should only exist to represent the internal mathematical state of the grid.

A. main.js (UI & Orchestration)

  • Initializes GameState, PerformanceMonitor, and LocalStats.
  • Handles DOM interactions (Clipboard paste events, pointer events for pan/zoom).
  • Parses raw Wordle clipboard text into a 2D binary grid.
  • Runs the requestAnimationFrame loop.
  • Binds the game_state.grid data to SVG/DIV elements using D3.js selections and transitions.

B. game-state.js (Pure Logic)

  • Maintains the 2D grid array representing live/dead cells.
  • Contains the core next_generation() logic representing Conway's rules.
  • Tracks historical grid hashes to detect when the simulation has stabilized (oscillating or frozen).
  • Exposes a clean API for main.js to read active cells, but never touches the DOM or window object.

C. local-stats.js (Persistence)

  • Interfaces with browser localStorage.
  • Hashes Wordle seeds to prevent double-counting statistics for the exact same pasted game.
  • Tracks high scores across sessions (e.g., longest-lived pattern, max mature gliders).

3. Directory & Component Hierarchy

The codebase is physically structured to separate highly reusable simulation components from app-specific business logic. This allows the generic components to be reused in future cellular automata projects.
