🛸 Glider Detection & Tracking Plan

Detecting a glider isn't just about matching a shape; it's about tracking a process through time. This plan outlines how we will move from a sparse grid to a collection of "identified" gliders.

1. The Detection Algorithm (Spatial)

Since we have live_cells, we don't scan the whole grid. Instead, we use each live cell as a potential constituent of a glider.

  • Step A: Iterate through live_cells.
  • Step B: For each cell, test it against the 4 phases (and 8 symmetries) defined in patterns.js.
  • Step C: If a match is found, tag those cells with a glider_id.

2. Identity Persistence (Temporal)

To avoid "counting" the same glider multiple times as it moves, we use Spatial Overlap Match.

Feature Mechanism
Unique ID New gliders get an incrementing_id.
Tracking In Gen N+1, look for a glider whose bounding box overlaps the bounding box of a glider from Gen N.
Displacement A glider moves at most 1 cell diagonally every 4 generations. We check a 3x3 search area around the previous center.

3. The "Party Parrot" Effect

Once a cell is identified as belonging to a Glider object, we override its CSS class in the renderer.

Rainbow Rotation

Cells mapped to glider_id: 42 will get the .party-glider class.

4. Implementation Steps

  1. Symmetry Generator: Write a utility to rotate/flip the patterns in patterns.js into all 8 valid orientations.
  2. The Scanner: In GameState.tick(), after apply_rules, run the detection pass.
  3. Glider Registry: Maintain a this.active_gliders map in GameState.
  4. D3 Update: Modify draw.js to check if a cell has a glider_id and apply the rainbow class.
