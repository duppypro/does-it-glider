Dynamic Grid Sizing Plan

To support gliders traveling indefinitely without hitting rigid borders, we are introducing a Dynamic Grid Sizing engine. It starts with a lightweight 128x128 layout and scales up by increments of 128 whenever cells near the boundary, preserving performance and memory for typical runs while supporting unbounded expansion when patterns take flight.

2. Architectural Strategy: Symmetric Symmetrical Padding

To grow the grid cleanly while supporting movement in all 4 directions, we propose Symmetric Symmetrical Padding. When a live cell breaches the boundary, we pad 64 dead cells on all four sides (Left, Right, Top, Bottom). This increases both width and height by 128 (e.g., from 128x128 to 256x256).

Symmetric Padding Mechanism

Breach!
Initial Grid (128x128)
Cell at (0, 0) triggers breach
➔
Expanded Grid (256x256)
All cells shifted by +64; old boundary is dashed

Mathematical Elegance: Shifting cell coordinates by +64 matches the SVG translation offset change +64 * CELL_PX exactly. The cells remain 100% visually stable on screen with zero jitter, without needing to adjust the camera zoom transform!

3. The Boundary Check

A boundary breach is detected inside the GameState.tick() method before we run the rules. We flag a breach if any live cell is within 1 cell of the edge:

  • cell.x <= 0 (Left breach)
  • cell.y <= 0 (Top breach)
  • cell.x >= grid_width - 1 (Right breach)
  • cell.y >= grid_height - 1 (Bottom breach)

Since a glider's neighborhood check looks 2 cells out, or Conway's neighbors look 1 cell out, ensuring a 1-cell safety margin is perfect to trigger resizing before any cell tries to access out-of-bounds indices.

4. Implementation Plan

Step A: Expose Grid Resizing in grid.js

Create update_grid_dimensions(w, h) which updates the underlying G_WIDTH, G_HEIGHT, adjusts the D3 background and graph-paper fill rects, and recalculates zoom translation constraints.

Step B: Implement Grid Expansion in GameState

Add _check_and_expand_grid() to game-state.js, called inside tick() before apply_rules_old_new(). If a breach is found:

  1. Calculate new dimensions (e.g. current_dimensions + 128).
  2. Allocate new grid_ping and grid_pong matrices.
  3. Copy old cells with an offset of +64 to the new matrices.
  4. Shift all coordinates in this.live_cells, this.next_live_cells_to_clear, and this.active_gliders by +64.
  5. Notify grid.js to resize the SVG canvas.
  6. Update the Bounding Box local stat to the new size.

Step C: Integrate Local Stats Tracking

Add MAX_BOUNDING_BOX_KEY to local-stats.js. Expose current and max stats in the UI layout inside main.js, aligning with the beautiful table theme.

5. Finalized Architecture & Memory Safety Ceiling

  • Memory Safety Ceiling (1024x1024 Max): To prevent escaping gliders from expanding the grid infinitely, we added a strict safety ceiling. If this.grid_width >= 1024, the grid stops expanding and standard Conway rules apply at the edge (allowing gliders to naturally hit the boundary and dissolve, keeping memory and CPU footprint completely bounded).
  • Strict Seed Constraints (5-wide x 6-high): All pasted clipboard scores and debug/diagnostic text seed injections are strictly truncated and validated to a maximum size of 5 columns wide and 6 rows high. This ensures the initial grid configuration is always neat, predictable, and runs at peak efficiency.
  • Automatic Resets on Load: Whenever a new seed is loaded, the grid size instantly resets back to 128x128. This allows each individual attempt to start in a clean, lightweight environment and only grow if the simulation demands it.

7. Clean Glider Escape (No Border Debris)

Normally, when a glider reaches the grid boundary (or the 1024x1024 ceiling), it breaks apart and leaves an ugly, static 2x2 block still-life debris on the border. To allow gliders to escape the universe cleanly without leaving debris, expanding the visible grid, or altering stats, we can implement an Active Glider Evaporator.

Proposed Implementation Mechanism:

  1. Escape Detection: In _detect_gliders(), we identify if any tracked glider (especially mature gliders) has its top-left anchor or any of its cells within 2 cells of the border:
    • glider.x <= 2 || glider.y <= 2
    • glider.x >= grid_width - 4 || glider.y >= grid_height - 4
  2. Clean Evaporation: Instead of letting the glider's leading cells get truncated (which destroys the pattern and leaves debris), we cleanly delete all cells belonging to that glider from this.live_cells, and clear them in both current_grid and next_grid.
  3. Suppressed Expansion: Because the glider is cleanly evaporated before the next generation tick's boundary breach check occurs, it never triggers a grid expansion or counts towards updating grid size stats!
  4. Thematic tracking: This allows us to track a new stat, e.g., Escaped Gliders, rewarding the player for successfully releasing gliders into the wild!
