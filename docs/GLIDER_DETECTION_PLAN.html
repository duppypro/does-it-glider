Glider Detection & Tracking Plan

Detecting a glider isn't just about matching a shape; it's about tracking a process through time. This plan outlines how we move from a sparse grid to a collection of "identified" gliders with persistent identity.

1. The Detection Algorithm (Spatial)

Since we have live_cells, we don't scan the whole grid. Instead, we use each live cell as a potential constituent of a glider.

  • Step A: Iterate through live_cells.
  • Step B: For each cell, test it against the base glider templates (and anchors) defined in patterns.js.
  • Step C: If a match is found and its cells aren't already claimed, tag those cells with a glider_id.

2. Identity Persistence (Temporal)

To avoid "counting" the same glider multiple times as it moves, we use Spatial Overlap Match anchored to the top-left coordinate.

Feature Mechanism
Unique ID New gliders get an incrementing_id.
Tracking & Displacement In Gen N+1, we look for a candidate whose top-left anchor coordinate is within a 5x5 area (dx <= 2 && dy <= 2) of a glider's anchor from Gen N, and whose orientation matches.
Maturation Newly detected gliders (age < 4) are considered Proto gliders. If a glider is destroyed or crashes before completing a full 4-generation cycle, it is recorded as a Tragic Fizzle.

3. The "Party Parrot" Effect

Once a cell is identified as belonging to a glider_id, it receives special rendering instructions.

Dynamic Programmatic Rainbow

Originally, this was implemented using CSS animations. However, because we use a Sparse DOM approach (where cells are created and destroyed dynamically as they move), the CSS animation would restart every time a cell was recreated, causing an unsightly "flash."

4. The "Solitary Glider" Problem

We are detecting processes, not just shapes. If other live cells are touching the glider, they act as "interference," causing the glider to crash or transform in the next generation. Therefore, a valid glider must exist in a sufficiently clear local neighborhood.

5. Sub-Agent Moat Analysis

To definitively answer whether a "strict moat" is necessary, a programmatic sub-agent was deployed during development (documented in moat-test.js).

Sub-Agent Findings & The Exception Bug:

  • Debris is inevitable: In ~100 configurations per phase, the expected 5 cells of the next glider phase survive, but the noise also survives or spawns new cells. This proves that a weak survival check is too permissive.
  • The Single Exception: For Phase 0 and Phase 2, there is exactly one live cell configuration (a single live cell at coordinate 0,0—the top left corner) that allows the glider to progress cleanly while not being completely empty.
  • Strict Empty Moat & Bug #25: Initially, the spec enforced a Strict Empty Moat (checking that all 20 non-glider cells in the 5x5 bounding box are dead). This caused a false negative on the known valid exception.
  • The Fix (Safe Noise): To resolve this, the code in game-state.js now allows a tiny, explicitly whitelisted set of safe_noise coordinates defined in patterns.js.
