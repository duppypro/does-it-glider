Decoupled Algorithmic & Visual Sizing Plan

To support gliders traveling indefinitely without triggering performance bottlenecks or artificial boundary reflections, we implement a decoupled sizing strategy: a very large 2048x2048 Algorithmic Grid and a dynamic Visual/Rendering Boundary that starts at 128x128 and expands only for non-glider activity.

1. Decoupled Sizing Philosophy

The universe array is statically modeled inside a massive 2048x2048 coordinate box. This box acts as the absolute boundary of physics. Live cells cannot exceed this range. However, rendering, panning, and zooming are strictly constrained to a much smaller Visual Grid (initially 128x128), centered around the loaded seed.

Concept Size/Bounds Role & Behavior
Algorithmic Grid 2048 x 2048 (Fixed) Calculates standard B3/S23 Conway rules. Gliders travel silently across this vast coordinate plane, invisible to the user once they leave the visual bounds. Assumes all coordinates outside this 2048 range are dead.
Visual Grid Starts at 128 x 128, expands by +128 D3 rendering, coordinate shifting, panning, and zooming are restricted to this box. It expands only if a non-glider cell breaches its current edge.

2. Dynamic Visual Sizing: Symmetrical Padding

To keep the screen completely stable during grid growth, we grow the visual box symmetrically. When a non-glider live cell breaches the visual boundary, we expand the visual bounds by 64 cells on all four sides (Left, Right, Top, Bottom). This increases both the visible width and height by 128 (e.g., from 128x128 to 256x256).

Performance Elegance: Because gliders traveling beyond the visual box do not trigger expansion, the visual board stays at 128x128 for typical runs. This limits DOM nodes and D3 joins to a tiny fraction of the universe, maintaining high framerates.

3. The Non-Glider Boundary Check

We perform the boundary check during GameState.tick(). A breach of the visual boundary is flagged if any live cell is outside the current visual bounds: cell.x < vis_left || cell.x >= vis_right || cell.y < vis_top || cell.y >= vis_bottom.

Crucial Rule: Before triggering visual grid growth, we check if the breaching cell belongs to any currently tracked active glider. If it is part of a glider, the breach is **ignored** and the visual bounds do not expand. The glider continues its flight into the outer void silently.

4. Filtered Stability Loop Detection

Normally, gliders flying into deep space would keep the simulation running infinitely, preventing the system from detecting a loop or static end-state. To resolve this:

  • We filter this.live_cells to compile a list of active cells that are either **inside the current visual boundary** OR are **non-glider cells** outside the boundary.
  • We generate the state history hash and check for Loops/Static cycles (up to 8 generations) using **only** this filtered set.
  • This ensures the simulation stabilizes and pauses the instant the core visible reaction settles, ignoring any invisible gliders soaring in deep space!

5. Clean Glider Escape (No Border Debris) & Visual-Border Tracking

When a glider reaches the absolute edge of the 2048x2048 universe, it would normally crash on the boundary and leave static 2x2 block debris. To prevent this, and to ensure the player is credited instantly when gliders soar off-screen, we maintain a dual-stage Active Glider Escape Engine:

  • Instant Visual-Border Escape Tracking: The moment all cells of a mature glider cross fully outside the current Visual Grid boundary, we mark the glider as officially escaped and increment the escaped_gliders_count. This guarantees the player is credited instantly, even if the core visible simulation stabilizes and pauses before the glider reaches the outer edge of the universe!
  • Background Physics & Edge Evaporation: After crossing the visual border, the glider continues its flight silently in the background of the 2048x2048 algorithmic grid. If it comes within 2 cells of the absolute 2048x2048 algorithmic grid boundary (i.e., x <= 2 || y <= 2 || x >= 2045 || y >= 2045), we cleanly evaporate all of its cells from both the grid buffers and live_cells so it does not leave border debris.

6. Interactive Study: Cosmic Glider Collisions

A fascinating consequence of computing glider physics beyond the visual boundary is the occurrence of Cosmic Collisions in the outer void. If a pattern launches multiple gliders with intersecting paths, they will collide in invisible space. We define two distinct behaviors for these collisions:

Scenario A: Complete Obliteration

Mechanism: Gliders collide head-on and eventually completely annihilate each other.

Boundary Impact: Even though the final state is total extinction, during the transition generations, the live cells momentarily fail to match a valid glider pattern. These intermediate chaotic frames represent "non-glider noise" outside the boundary, which **will trigger a visual expansion** to reveal the collision site to the player before it fades away.

Scenario B: Debris Creation

Mechanism: Gliders collide at a specific phase offset and fuse into a stable, static 2x2 block.

Boundary Impact: As soon as the gliders merge, they break up and cease to be recognized as active gliders. The resulting static debris represents a permanent non-glider breach, **permanently expanding the visual grid** to swallow the collision site.

Philosophical Design Decision: We explicitly accept and preserve this behavior. Even though intermediate chaos momentarily enlarges the board, outer-void glider collisions are rare, magical phenomena. Treating them as expansion-triggering events ensures the player discovers them organically. This lays the groundwork for recording statistics that may unlock unique **Space Archaeology achievements** or rewards in the future!
